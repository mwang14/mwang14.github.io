First vulnhub VM(Tr0ll) writeup
July 8, 2016

The hardest part of compromising this machine was by far the setup. I first tried the kioptrix machine, but wasn't able to configure the network configuration properly. However, after two days of finding a suitable machine and setting up Kali, a day and a half of tinkering with the machine, I've finished my first vulnhub VM.

So after booting up the VM, I went to the IP in my local browser. I didn't know what I expected, given the name of the VM...

The "secret" page didn't help much either.

The next step I took was nmap'ing the IP to see what services they were running. They were running 3(ftp, ssh, and http). I've had the most experience with FTP, so I tried that one first. It was running VSFTPD, so no known vulnerabilities there. However, they allowed anonymous access!

I found one file(lol.pcap), and downloaded it.

The file was quite a...troll. However, it did give a hint "you almost found the sup3rs3cr3tdirlol". Turns out that that was another directory on the website.

I downloaded the file(roflmao) and examined it. It had one more hint: "check the address 0x0856BF to proceed."

My first thought here was that it was some hardware address that I would have to exploit using some sort of remote buffer overflow. (Un)luckily, I still don't know how to do that, so I tried some other options. I looked the address up on Google Maps, and found it was a street named "Recreation" in Singapore, and thought it could be the password to the root account. It wasn't. I then tried to put the address into the URL again, and it revealed yet another directory.

I downloaded the files contained in the folders(good_luck contained a text file with usernames, and this_folder_contains_the_password had a text file called "Pass.txt" that said "good job :)"), and fired up Hydra. Unfortunately, it seemed the machine had fail2ban, so every time Hydra failed I had to restart the machine, which was inconvenient. It seemed that the password file was wrong. After playing around, I decided to try "Pass.txt" as the password. I'd like to say that I figured that out by the name of the parent directory("this_folder_contains_the_password"), but honestly... I just guessed :P. I now had a shell inside the machine, albeit without root privileges.

After having a shell, I used the find command("find / -writable -executable" was the exact command) to search for any interesting files I could modify. I found one file that stuck out in particular: /lib/log/cleaner.py.

After examining the file, I realized it was owned by root, and I could write to it!

I modified the file so that I would get root access by running another file in tmp.

Finally, I went into the root directory, and found the flag!
